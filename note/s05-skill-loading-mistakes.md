# Skill Loading — 错误点与模糊点笔记

聚焦自己卡住的地方，而不是课程全部内容。目标是下次遇到同类问题能直接拿出论据。

---

## 1. Q1 — 一句话答案只说了"问题"，没说"解法"

**我答了什么：**
> 注意力问题，核心问题甚至不是 token，而是注意力，是上下文的污染。

**漏了什么：**
只描述了"为什么是个问题"，没说"我们怎么解"。完整的一句话答案必须同时包含问题和解法：

> SkillLoader 把 skill 拆成「目录常驻 + 详情按需加载」两层，避免把所有专业知识塞进 system prompt 污染模型注意力。
> └─────────── 解法 ───────────┘└─────────── 问题 ─────────┘

**教训：** 问题陈述 + 解决思路 = 一句话答案的最小单元。缺任何一半，听众都接不住。

## 2. Q2 追问没答 — 热路径 vs 冷路径

**追问：**
> 为什么我们故意选了启动时扫盘，而不是每次调用时扫盘？

**该这样答：**

| 路径 | 频率 | 例子 |
|------|------|------|
| 冷路径（cold path） | 低频 | 启动初始化、配置加载 |
| 热路径（hot path） | 高频 | 每次工具调用、每次循环迭代 |

优化原则：热路径每微秒都重要；冷路径只要"够快到用户感知不到"就行。

`readdirSync` + N 次 `readFileSync` 放在启动时：
- 启动慢 5–10ms → 用户感知不到（冷路径）
- 之后每次 `getContent()` 是 O(1) Map 查找 → 零额外 IO（热路径快）

反过来"每次调用扫盘"：每次 `load_skill` 都触发 N 次磁盘 IO，工具响应慢且无谓。

**教训：** 评估"该不该缓存 / 该不该预加载"时，先问：这是热路径还是冷路径？

## 3. Q3 — 边界/职责讲不硬，循环论证

**我答了什么：**
> 不可以，这样使得边界不清晰，假如这次这么做了，那以后还要怎么改 skillLoader，他的职责就不清晰了，不清晰的职责会让职责越来越不清晰。

**问题：循环论证。** "职责不清晰会让职责越来越不清晰"——用结果当原因。同事反问"为什么不清晰是问题？"就接不住了。

**真正的反驳论据（按硬度排序）：**

**A. 复用性（最硬）**
SkillLoader 是通用模块。如果未来写 CLI 命令、生成文档站、写自动化测试验证 skill 格式——这些场景都不需要 LLM wrapper。一旦 `getContent()` 强行包 `<skill>` 标签，所有非 LLM 用法都要先 strip 再处理。

**B. 厂商格式锁定**
今天是 `<skill name="...">...</skill>`，明天 OpenAI 推荐 markdown fence，后天 Gemini 要 JSON。输出格式属于调用方的渲染层，不属于数据加载层。

**C. 测试性**
- 现在断言 `getContent("pdf").includes("pdftotext")` 即可
- 改了之后断言里要写完整的 `<skill name="pdf">\n...\n</skill>`——耦合实现细节，标签格式一改测试全挂

**D. DRY 不是这个用法**
wrapper 现在只在 `toolRuntime.ts` 一处出现。一处出现的代码不叫"重复"，叫"使用"。如果真有 N 处需要包，正确解法是在调用方所在层写 helper，不是塞进 SkillLoader。

**教训：三把尺子检查任何 PR**
1. 换个调用方还能用吗？（复用性）
2. 这两块东西变化频率一样吗？（生命周期）
3. 这是业务关注点还是横切关注点？（统计/日志/缓存属于横切）

## 4. Q4 — 选对方案但论据空泛

**我答了什么：**
> 选方案 B，我想保证 skillLoader 的职责明确，他只负责加载，他就是加载器，由中间层来负责记录频次。虽然 A 并没有影响原本的逻辑……我说不下去了。

**问题：** 直觉对，但讲不硬。"我说不下去了"是诚实的自我评价。

**真正的反驳论据：**

**1. 双计数污染**
未来如果 CLI、文档生成器、单元测试都调用 `getContent`，这些非真实用户的调用会污染统计。把统计塞进 loader = 它分不清"真用户"和"工具调用"。

**2. 生命周期不同**

| 关注点 | 何时变化 |
|--------|---------|
| 加载（loader） | 启动一次，之后不变 |
| 使用统计（counter） | 每次调用都变 |

变化频率不同的状态混在一类里，未来要持久化、跨进程聚合、定期重置时全是麻烦。

**3. 测试相互污染**
测 loader 时要清理 counter；测统计时要 mock loader——本来无关的两件事互相牵扯。

**4. 横切关注点**
统计、日志、监控、tracing 都属于"和业务无关但每个业务都要"的东西。正确做法是在调用边界处理（中间件/装饰器/钩子），不是塞进业务模块。

**我没想到的最优解：方案 C（装饰器模式）**

```ts
function withUsageTracking(loader: SkillLoader) {
  const counts = new Map<string, number>();
  return {
    getContent(name: string) {
      counts.set(name, (counts.get(name) ?? 0) + 1);
      return loader.getContent(name);
    },
    getStats: () => counts,
  };
}
```

- SkillLoader 一行不动
- 调用方无需记账（自动透明）
- 想换实现/关掉统计 → 换一行包装就行

同源思想：React HOC、Express middleware、Python `@decorator`。

## 5. Q5 — 选对方案但理由太软

**我答了什么：**
> 我选 B，这个语义我喜欢，更清晰。

**问题：** "喜欢"/"清晰"是直觉，不是工程判据。会被人用同样的"我喜欢 A"反驳掉。

**客观判据：搜索功能有没有内部状态？**

| 搜索算法 | 有状态吗 | 推荐方案 |
|---------|---------|---------|
| 关键词 substring 匹配 | 无（每次现算） | C 独立函数最优 |
| Tag 倒排索引 | 有（构造一次复用） | B 类持有索引 |
| Embedding 语义搜索 | 强状态（预计算 embeddings） | B 必须 |

**B 的真理由：** 不是"语义清晰"，是"为 embedding 留空间"——预计算的索引必须有地方住，函数无法持有状态。

**各方案的真代价：**
- **A 的代价：** 类型膨胀传染，最终变成 God Object（10 个方法还行，30 个就没人敢动）
- **B 的代价：** 引入类间依赖图。约定依赖方向必须单向：`SkillSearch → SkillLoader`，loader 永远不知道 search 存在
- **C 的代价：** 无法持有状态，embedding 索引每次重建

## 6. 隐性盲区（概念混淆）

### 6.1 "tools 是我拼进 SYSTEM 的"——错误前提

❌ 我以为：先 skill 后 tool 是字符串拼接顺序
✅ 实际：`system` 和 `tools` 是 API 的两个独立字段，SDK 内部拼，位置我控制不了。我能控制的只有 `system` 字符串内部顺序。

### 6.2 "subagent 的 prompt 是父 agent 传的"——概念混淆

| 字段 | 来源 |
|------|------|
| `system` | 共享的 SYSTEM 常量（主/子 agent 都一样） |
| `messages[0]` (user) | 父 agent 调 `task(prompt)` 传的任务 |

`task` prompt 只是 user message，不是 system prompt。

## 7. 词汇表（能用但叫不出名的概念）

| 术语 | 含义 | 应用 |
|------|------|------|
| 装饰器模式 (Decorator Pattern) | 用包装器在不改原对象的前提下加功能 | Q4 的最优解 |
| 横切关注点 (Crosscutting Concern) | 统计/日志/缓存等"业务无关但人人需要"的关注点 | Q3、Q4 反驳论据 |
| God Object | 什么都干、什么都不擅长的巨型类 | Q5 反驳 A 方案 |
| 依赖倒置 (DIP) | 高层模块不依赖低层，依赖方向单向 | Q5 选 B 时的约束 |
| 热路径 / 冷路径 | 高频/低频执行代码，优化策略不同 | Q2 启动时扫盘的理由 |
| 注意力稀释 / lost-in-the-middle | 长 prompt 中段权重最低 | s05 注入顺序问题 |
| Prompt prefix caching | LLM 厂商对前缀完全一致的 prompt 缓存命中折扣 | 为什么 SYSTEM 用 const 而非 function |

## 8. 自检 Checklist

写代码或评 PR 时，强制问自己：

```
□ 换个调用方还能用吗？               ← 检查复用性
□ 这两块东西变化频率一样吗？         ← 检查生命周期混杂
□ 这是业务关注点还是横切关注点？     ← 决定塞业务模块还是用装饰器/中间件
□ 这是热路径还是冷路径？             ← 决定预算/缓存策略
□ 这个判断是基于客观判据还是直觉？   ← 强迫自己说出可证伪的标准
```

80% 的"看似合理但其实错"的设计，能被这 5 个问题挡掉。
